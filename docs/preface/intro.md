在你翻开正文之前，我想先和你聊聊一个容易被忽略的话题：**偏见**。

如果你问一个十年前的程序员：“Python 是什么？”，他很可能会这样回答：  
“写脚本很快，但跑得不快；类型松散，适合原型，不太适合严肃工程。”

这种看法并非空穴来风——它来自一个真实存在过的 Python 时代。

但如果你问今天的后端工程师、数据工程师，或 AI 领域的开发者同样的问题，他也许会向你展示另一幅图景：  
使用 FastAPI 构建的高并发 API，借助 Pydantic 明确约束数据边界，依赖完善的类型注解进行静态检查，并由现代工具链保障代码质量与交付效率。

**这两种描述，指向的都是 Python，却并不是同一个 Python。**

## 并不是所有的 Python 都长得一样

Python 是一门正在持续演进的语言。

尤其自 Python 3.5 引入类型注解（Type Hints）以来，到 Python 3.12 泛型语法逐步成熟，Python 的工程实践已经发生了深刻变化。

从结果上看，它逐渐分化出了两种截然不同的使用方式：

1. **以脚本为中心的 Python**：  
   强调快速试错、即写即用，适合一次性任务和探索性工作。

2. **以工程为中心的 Python**：  
   重视类型、结构和边界，强调可维护性、可测试性和长期演进能力。

这本书，关注的是**后者**。

这并不意味着前者是“错误的”，而是意味着：  
当你希望用 Python 构建长期存在的软件系统时，需要一套不同的思维方式。

## 为什么写这本书？

现有的 Python 学习资料，大致存在两个常见的问题。

### 1. 以“零基础”为唯一假设

很多教程默认读者从未接触过编程，因此花费大量篇幅解释变量、条件判断和循环结构。

这对完全初学者来说是必要的，但如果你已经学过其他语言——哪怕只是一点点——这些内容很快会变成负担。

你真正需要的，不是重新学习“如何编程”，而是理解：  
**Python 是如何看待问题的，它鼓励你如何组织代码。**

### 2. 停留在语法层面

另一类教程，则止步于“语法正确”。

它们会教你如何定义类，却很少讨论什么时候不该使用继承；  
教你如何导入模块，却回避循环依赖背后的设计问题；  
教你并发，却不解释运行时模型的限制与代价。

而工程能力，恰恰藏在这些“为什么”里。

## 这本书的定位

这不是一本面面俱到的语言手册（官方文档已经非常出色），  
而是一本关于**如何写好 Python 代码**的实践指南。

- 你不需要任何 Python 经验；
- 如果你来自其他语言，这本书会帮助你完成心智模型的迁移；
- 如果你是第一次接触编程，它也会引导你从一开始就站在正确的工程起点上。

本书统一基于 **Python 3.12**，并**系统性地使用类型注解**作为表达与约束工具。

## 阅读方式与核心原则

贯穿全书，我们将反复强调三件事：

1. **类型是一种沟通方式**  
   类型注解不仅服务于工具，更服务于人。清晰的类型，能让代码自己说明意图。

2. **边界比逻辑更重要**  
   数据一旦进入系统，就应该是可信的。我们将重点讨论如何在 IO 边界建立防线，而不是在内部到处补救。

3. **理解抽象的代价**  
   Python 提供了大量强大的抽象机制，但每一种抽象都有运行时成本与认知成本。理解它们，才能做出理性的取舍。

## 写在最后

我希望当你合上这本书时，  
你写下的 Python 代码，不再依赖模糊的约定和隐式的假设，  
而是具备清晰的结构、明确的边界，以及经得起时间考验的可读性。

Python 是一门允许你不断靠近这种状态的语言。

让我们带着**新的目光**，重新认识它。
